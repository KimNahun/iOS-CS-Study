# 1️⃣ 프로세스 / 스레드

프로세스와 스레드는 운영체제에서 프로그램이 실행되는 방식을 이해하기 위한 핵심 개념입니다.

이 둘을 이해하기 위해서는 먼저 **프로그램**이라는 개념을 알 필요가 있습니다.

## 프로그램

> 실행되지 않은 상태의 코드 파일
> 

우리가 작성한 코드가 디스크에 저장되어 있는 상태로, 아직 메모리에 올라가있지 않은 **정적인 상태**입니다.

이 단계에서는 아직 운영체제가 프로그램에게 독립적인 메모리 공간을 할당해주지 않습니다. CPU도 사용하지 않고, 메모리 공간도 차지하지 않으며 단순히 실행을 기다리고 있는 상태일 뿐입니다.

모든 프로그램은 운영체제가 실행되기 위한 메모리 공간을 할당해줘야 실행될 수 있기 때문에, 이 상태에서는 실행되지 않고 가만히 있다는 뜻입니다.

즉, 프로그램은 말 그대로 **‘코드 덩어리’**이며 실행되기 전 단계의 개념이라고 볼 수 있습니다.

## 프로세스

> 운영체제로부터 자원을 할당받은 **작업**의 단위
> 

<img width="1444" height="624" alt="image" src="https://github.com/user-attachments/assets/e49dff2b-9f35-4b5b-a871-681a0161436f" />

프로그램을 실행하면 운영체제는 해당 프로그램을 메모리에 올리고, 실행에 필요한 자원을 할당합니다. 이 순간 프로그램은 정적인 상태에서 동적인 상태로 바뀌게 되며, 이를 **프로세스**라고 합니다.

프로세스는 운영체제로부터 메모리, CPU 시간, 파일 등의 자원을 할당받아 실행되는 작업의 단위입니다. 각 프로세스는 독립된 메모리 공간을 가지며, 다른 프로세스의 영역에 직접 접근할 수 없습니다.

이러한 독립성 덕분에 하나의 프로세스에서 문제가 발생하더라도 다른 프로세스에는 영향을 주지 않는다는 장점이 있습니다. 하지만, 그만큼 생성 비용이 크고 프로세스 간 데이터를 주고받기 위해서는 별도의 통신 방식이 필요하다는 단점도 존재합니다.

## 스레드

> 프로세스 내부에서 실제 작업을 수행하는 실행 흐름의 단위
> 

하나의 프로세스는 여러 개의 스레드를 가질 수 있으며, 이 스레드들은 프로세스가 할당받은 자원을 공유하면서 동시에 작업을 수행합니다.

스레드는 프로세스와 달리 메모리를 공유하기 때문에 데이터 접근이 빠르고 효율적입니다. 예를 들어 하나의 프로그램에서 화면을 출력하는 작업과 데이터를 처리하는 작업을 동시에 수행할 수 있는 이유가 바로 스레드 때문입니다.

하지만 자원을 공유하는 구조이기 때문에 여러 스레드가 동시에 같은 데이터에 접근할 경우 동기화 문제가 발생할 수 있습니다. 또한 하나의 스레드에서 오류가 발생하면 같은 프로세스 내 다른 스레드에도 영향을 줄 수 있다는 점에서 안정성은 상대적으로 낮습니다.

- 🍎 iOS와 연결
    
    ### 1. iOS 앱은 프로세스와 스레드를 어떻게 사용할까?
    
    - iOS에서 하나의 앱은 하나의 프로세스인가?
    - 여러 앱은 서로 영향을 줄 수 있는가?
    
    iOS에서는 하나의 앱 = 하나의 프로세스로 실행됩니다.
    
    각 앱은 샌드박스 환경에서 동작하기 때문에, 다른 앱의 메모리에 접근할 수 없습니다.
    
    ### 2. iOS에서 스레드는 어떻게 활용될까?
    
    - UI는 왜 메인 스레드에서만 동작할까?
    - 백그라운드 작업은 왜 따로 처리할까?
    
    iOS에서는 하나의 프로세스 내부에 여러 스레드가 존재합니다.
    
    - 메인 스레드 → UI 처리
    - 백그라운드 스레드 → 네트워크, 파일 처리
    
    여러 스레드를 사용하는 이유는 동시에 여러 작업을 처리하기 위해서입니다.
    
    ### 3. 왜 UI는 반드시 메인 스레드에서만 동작할까?
    
    UI는 공유 자원입니다. 여러 스레드가 동시에 접근하면 충돌이 발생합니다.
    
    그래서 iOS는 UI 작업을 메인 스레드로 강제합니다.
    
    만약 UI 작업이 메인 스레드에서 그려지지 않는다고 가정해보면 각 작업은 서로 다른 스레드에서 진행될 것이고 그 작업이 끝나는 시간에 따라 각각 UI가 반영이 될 것입니다.
    
    그렇게 되면 그림을 계속 다시 그리고, 다시 그리고… 각 스레드마다 RunLoop를 가지고 있다보니 모아서 한 번에 반영을 할 수가 없는 상황이 됩니다.
    
    ### 4. iOS에서 멀티 스레딩은 어떻게 구현할까?
    
    - GCD(Grand Central Dispatch)
    - OperationQueue
    - Swift Concurrency(async/await)
    
    iOS는 개발자가 직접 스레드를 다루기보다는 큐 기반으로 작업을 관리한다.

# 2️⃣ 컨텍스트 스위칭

> CPU가 현재 실행 중인 작업을 멈추고, 다른 작업으로 전환하는 과정
> 

## 컨텍스트 스위칭이 필요한 이유

CPU는 한 번에 하나의 작업만 실행할 수 있습니다.

하지만 우리는 여러 프로그램이 동시에 실행되는 것처럼 느낍니다. 예를 들어 카카오톡을 하면서 음악을 듣고, 동시에 유튜브를 시청하는 것도 자연스럽게 가능합니다.

이는 실제로 여러 작업이 동시에 실행되는 것이 아니라, CPU가 매우 빠르게 작업을 번갈아가며 실행하기 때문입니다. 이러한 전환 과정이 바로 컨텍스트 스위칭입니다.

## 컨텍스트란?

> 작업의 현재 실행 상태를 나타내는 정보
> 

컨텍스트는 작업을 중단했다가 다시 실행할 때, 이전 상태를 그대로 이어서 수행할 수 있도록 해주는 정보입니다.

대표적으로 다음과 같은 정보들이 포함됩니다.

- 프로그램 카운터 (Program Counter, PC)
- CPU 레지스터 값
- 스택 정보
- 메모리 상태

즉, 컨텍스트는 “이 작업을 어디까지 수행했는지”를 나타내는 모든 정보라고 볼 수 있습니다.

## 컨텍스트 스위칭은 어떻게 진행되는가?

운영체제는 각 작업의 상태를 **PCB(Process Control Block)**를 통해 관리합니다.

컨텍스트 스위칭은 다음과 같은 과정으로 이루어집니다.

<aside>

1. 현재 실행 중인 작업의 상태(레지스터 값 등)를 PCB에 저장한다.
2. 다음에 실행할 작업의 PCB에서 상태 정보를 읽어온다.
3. 해당 정보를 CPU 레지스터에 복원한다.
4. CPU는 해당 작업을 이어서 실행한다.
</aside>

이 과정을 통해 CPU는 여러 작업을 번갈아 실행하면서도, 각 작업이 끊기지 않고 이어서 수행되는 것처럼 보이게 합니다.

## 프로세스 VS 스레드 컨텍스트 스위칭

컨텍스트 스위칭은 프로세스 단위와 스레드 단위에서 모두 발생할 수 있으며, 그 비용에는 차이가 있습니다.

### 프로세스 컨텍스트 스위칭

- 서로 다른 메모리 공간을 사용
- 주소 공간, 메모리 매핑 등 교체 필요

⇒ 비용이 크고 무겁습니다.

### 스레드 컨텍스트 스위칭

- 동일한 프로세스 내부에서 실행
- 메모리 공간을 공유

⇒ 비교적 비용이 적고 가볍다.

## 컨텍스트 스위칭 비용

컨텍스트 스위칭은 필수적인 과정이지만, 그 자체로는 실제 작업을 수행하지 않습니다.

즉, 컨텍스트 스위칭이 과도하게 발생하면 CPU는 작업을 처리하는 시간보다 작업을 전환하는 데 더 많은 시간을 사용하게 됩니다. 이는 전체적인 성능 저하로 이어질 수 있습니다.

## iOS와 컨텍스트 스위칭

iOS에서도 여러 작업이 동시에 실행되는 것처럼 보이지만, 실제로는 CPU가 스레드를 빠르게 전환하면서 이를 처리합니다.

Swift Concurrency에서는 다음과 같은 코드를 통해 실행 흐름을 제어할 수 있습니다.

```swift
Task.yield()
```

> 현재 작업을 일시 중단하고 다른 작업이 실행되도록 합니다.
> 

[yield() 공식문서](https://developer.apple.com/documentation/swift/task/yield())

`yield()`는 현재 작업의 실행을 잠시 양보하여, 다른 작업이 실행될 수 있도록 힌트를 제공하는 역할을 합니다. 즉, 장시간 실행되는 작업이 CPU를 독점하지 않도록 스스로 실행을 잠시 중단하는 것입니다.

다만, 해당 작업의 우선순위가 가장 높은 경우에는 즉시 다시 실행될 수 있기 때문에 이 방법이 항상 컨텍스트 스위칭을 보장하는 것은 아닙니다.

결과적으로 컨텍스트 스위칭이 과도하게 발생하면 성능이 저하될 수 있기 때문에, iOS에서는 Grand Central Dispatch와 같은 메커니즘을 통해 스레드를 효율적으로 관리하고 불필요한 전환을 최소화합니다.

# 3️⃣ 동기 / 비동기

## 동기(Synchronous)

> 작업이 순서대로 실행되며, 이전 작업이 완료될 때까지 다음 작업이 시작되지 않는 방식
> 

동기 방식에서는 현재 실행 중인 작업이 끝날 때까지 다음 작업은 대기합니다.

즉, 실행 흐름이 끊기지 않고 **직렬적으로 이어지는 구조**입니다.

<img width="914" height="285" alt="image" src="https://github.com/user-attachments/assets/ab9effd8-7a5b-4ea1-a608-bec3e3a3a50d" />

task1을 스레드 2로 보냈지만, 메인 스레드에는 여전히 block이란 것이 막고 있어서 task2가 실행되지 않고 있습니다.

즉, 하나의 작업을 다른 스레드로 보내더라도 그 작업이 완료될 때까지 현재 스레드는 다음 작업을 수행하지 못하고 기다리게 됩니다.

```swift
DispatchQueue.global().sync {
	print("Hello")
	print("World")
	print("!")
}
```

위 코드는 글로벌 큐에서 작업을 실행하지만, `sync`를 사용했기 때문에 해당 작업이 모두 끝날 때까지 현재 스레드는 대기합니다.

즉, 세 개의 `print`가 모두 실행된 이후에야 다음 코드가 실행된다.

## 비동기(Asynchronous)

> 작업을 요청한 후, 결과를 기다리지 않고 다음 작업을 바로 실행하는 방식
> 

비동기 방식에서는 특정 작업을 다른 스레드에 위임한 뒤, 그 작업이 끝나는 것을 기다리지 않고 현재 흐름을 계속 진행합니다.

<img width="914" height="285" alt="image" src="https://github.com/user-attachments/assets/c6f1d521-010e-4528-bca7-d25cf75e10be" />

<img width="920" height="300" alt="image" src="https://github.com/user-attachments/assets/ad2b18a1-5650-4c93-aab3-5c8b6abe2f73" />

사진의 예시에서는 task1이 다른 스레드로 옮겨갔고 메인 스레드에서는 task2가 task1의 작업을 기다리지 않고 즉시 실행됩니다.

```swift
DispatchQueue.global().async {
	print("Hello")
	print("World")
	print("!")
}
```

위 코드는 작업을 백그라운드 큐에 맡기고, 현재 스레드는 즉시 다음 작업을 수행합니다.

즉, `print`가 언제 끝나는지는 신경 쓰지 않고 프로그램은 계속 진행됩니다.

이처럼 비동기는 작업의 완료 시점과 현재 실행 흐름이 분리되어 있다는 특징을 가집니다.

### DispatchQueue

> 작업을 동기 또는 비동기로 실행할 수 있게 해주는 객체
> 
- sync ⇒ 작업이 끝날 때까지 기다림
- async ⇒ 작업을 맡기고 바로 다음 코드 실행

### Swift와 비동기 프로그래밍

#### **1. Callback 기반 비동기**

함수나 클로저를 전달하여, 작업이 완료된 시점에 결과를 처리하는 방식입니다

```swift
func fetchData(completion: @escaping (String) -> Void) {
    DispatchQueue.global().async {
        let result = "데이터 로드 완료"
        completion(result)
    }
}
```

### **2. Swift Concurrency (async/await)**

최근에는 Swift의 비동기 처리 방식으로 `async/await`이 사용됩니다.

```swift
func fetchData() async -> String {
    return "데이터 로드 완료"
}

Task {
    let result = await fetchData()
    print(result)
}
```

### **3. Grand Central Dispatch (GCD)**

iOS에서 가장 기본이 되는 비동기 처리 방식으로, 큐(Queue)를 기반으로 작업을 관리합니다.

개발자는 직접 스레드를 생성하거나 관리하지 않고, 작업을 큐에 넣는 방식으로 병렬 처리를 수행할 수 있습니다.

```swift
DispatchQueue.global().async {
    print("백그라운드에서 실행")
}

DispatchQueue.main.async {
    print("메인 스레드에서 UI 업데이트")
}
```

### 4. Combine

Apple에서 제공하는 선언형 비동기 처리 프레임워크로, 데이터의 흐름과 변화를 중심으로 작업을 처리합니다.

```swift
import Combine

let publisher = Just("Hello")
publisher
    .sink { value in
        print(value)
    }
```

### 5. RxSwift

ReactiveX 기반의 라이브러리로, 비동기 이벤트를 스트림 형태로 처리하는 반응형 프로그래밍을 지원합니다.

```swift
import RxSwift

let observable = Observable.just("Hello")
observable.subscribe(onNext: { value in
    print(value)
})
```

# 4️⃣ 직렬 / 병렬

동시성, 직렬, 병렬은 모두 작업을 처리하는 방식에 대한 개념이지만, 서로 다른 기준에서 구분된다.

이 개념들을 정확히 이해하려면 먼저 **동시성(Concurrency)** 부터 살펴보는 것이 중요하다.

## 동시성 프로그래밍

> 여러 작업을 **동시에 실행되는 것처럼 보이게 처리하는 방식**
> 

<img width="1888" height="892" alt="image" src="https://github.com/user-attachments/assets/611734a8-30d0-4290-adb6-1a0bd0c79209" />

여기서 중요한 점은 ‘동시에’라는 표현이 실제로 같은 순간에 실행된다는 의미가 아니라, **CPU가 매우 빠르게 작업을 번갈아 실행한다는 것**입니다.

즉, 하나의 CPU(코어)에서 여러 스레드가 번갈아가며 실행되며, 이 과정에서 **컨텍스트 스위칭**이 발생합니다.

## 직렬 프로그래밍

> 하나의 작업이 끝난 후 다음 작업을 실행하는 방식
> 

동시성 프로그래밍의 반대되는 개념은 병렬 프로그래밍이 아니라 **직렬 프로그래밍**입니다.

동시성 프로그래밍은 다중 스레드를 활용하는 것이지만, 직렬 프로그래밍은 단 하나의 스레드에서만 작업하는 것입니다.

즉, 직렬 프로그래밍은 **작업들이 순차적으로 처리되는 구조**이며, 한 번에 하나의 작업만 수행됩니다.

```swift
Task1 → Task2 → Task3
```

직렬 방식은 효율이 떨어질 수 있지만, **작업의 순서가 중요한 경우에는 반드시 필요합니다.**

**예시**

- 파일 읽기 → 데이터 처리 → 결과 저장
- 로그인 → 인증 → 데이터 요청

## 병렬 프로그래밍

> 여러 작업을 **실제로 동시에 실행하는 방식**
> 

<img width="1930" height="900" alt="image" src="https://github.com/user-attachments/assets/4334b41d-8ff9-4195-ac84-f587f6a5bca1" />

이는 하나의 CPU가 아니라, **여러 개의 코어(CPU)**를 사용하여 작업을 동시에 수행한다는 의미입니다.

핵심은 물리적으로 동시에 실행되고, 멀티코어 환경에서만 가능하다는 것입니다.

동시성과 달리, 병렬은 실제로 같은 시점에 여러 작업이 실행됩니다.

#### 동시성 vs 병렬

| **구분** | **동시성** | **병렬** |
| --- | --- | --- |
| **실행방식** | 번갈아 실행 | 동시에 실행 |
| **CPU** | 하나로도 가능 | 여러 개 필요 |
| **개념** | 논리적 | 물리적 |

# 5️⃣ Race Condition

> 여러 Thread가 하나의 자원에 동시에 접근해서 자원이 변질되어 원하는 결과가 나오지 않는 현상
> 

비동기적으로 코드가 실행되면 여러 Thread가 동시에 하나의 자원에 접근하는 경우가 생깁니다.

이때 Thread가 한 자원 접근해 해당 자원에 변화를 주게 되면, Race Condition 발생되고, 원하는 결과를 얻기 힘들어진다.

```swift
import Foundation

var counter = 0

let queue = DispatchQueue.global()
let group = DispatchGroup()

for _ in 0..<1000 {
    group.enter()
    
    queue.async {
        counter += 1
        group.leave()
    }
}

group.notify(queue: .main) {
    print("최종 값:", counter)
}
```

**기대하는 결과**

<aside>

최종 값: 1000

</aside>

**실제로 나올 수 있는 결과**

<aside>

최종 값: 932
최종 값: 857
최종 값: 1000 (운 좋으면)

</aside>

`counter += 1`은 counter 값을 읽음 → +1 수행 → 다시 저장 순서로 동작하게 됩니다.

하지만, 여러 스레드가 동시에 같은 값을 읽고, 동시에 수정하고, 덮어씌운다면 일부 +1 연산이 누락되게 됩니다.

이러한 현상을 Race Condition이라고 하는데요,
Race Condition이 발생하는 조건은 다음과 같습니다.

- 공유 자원
- 여러 스레드
- 동시에 접근
- 순서 보장 없음

이 같은 문제를 해결하기 위해서는 **Thread Safe**가 필요합니다.

## Thread Safe

> 여러 스레드가 동시에 하나의 자원에 접근하더라도, 데이터의 일관성과 정확성이 깨지지 않도록 보장하는 것
> 

Race Condition이 발생하는 이유는 여러 스레드가 하나의 자원을 동시에 수정하면서, 실행 순서가 보장되지 않기 때문입니다.

Thread Safe란 이러한 상황에서도 **데이터가 의도한 대로 유지되도록 보호하는 것**을 의미합니다.

즉, 여러 스레드가 동시에 접근하더라도 값이 꼬이지 않고 연산이 누락되지 않으며 항상 동일한 결과를 보장하는 상태를 만드는 것입니다.

- DispatchSepaphore ⇒ 동시에 접근 가능한 스레드의 수를 제한
- Custom Serial Queue ⇒ 한 번에 하나의 작업만 실행하도록 보장
- NSLock ⇒ 특정 코드 영역을 잠궈 한 번에 하나의 스레드만 접근
- Dispatch Barrier ⇒ 특정 시점에 단독으로 작업 실행 보장

# 6️⃣ Deadlock

> 두 개 이상의 스레드 또는 프로세스가 서로가 가진 자원을 기다리며, 영원히 작업이 진행되지 않는 상태
> 

Deadlock(교착 상태)은 멀티스레드 환경에서 발생할 수 있는 대표적인 문제로, 서로가 서로를 기다리기 때문에 **어떠한 작업도 더 이상 진행되지 않는 상태**를 의미합니다.

iOS에서는 특히 `DispatchQueue.main.sync`를 잘못 사용할 경우 Deadlock이 쉽게 발생할 수 있습니다.

메인 큐는 **Serial Queue(직렬 큐)**이기 때문에 한 번에 하나의 작업만 실행할 수 있습니다.

즉, 현재 실행 중인 작업이 끝나야 다음 작업이 실행됩니다.

```swift
// 메인 스레드에서 실행 시 데드락 발생
DispatchQueue.main.sync {
    print("이 코드는 실행되지 않습니다.")
}
```

이 경우에는 메인 스레드가 `DispatchQueue.main.sync` 를 호출합니다. 메인 스레드는 sync의 내부 블록이 끝나기를 기다리는데, 정작 그 블록은 메인 큐에서 실행되어야 하므로 서로 대기하게 됩니다.

### Deadlock 해결 방법

- **비동기 처리:** `sync` 대신 `async`를 사용하여 현재 큐의 작업을 중단하지 않도록 합니다.
- **큐 분리:** 메인 큐가 아닌 다른 Concurrent Queue나 별도의 Serial Queue를 사용하여 메인 스레드와 분리합니다. 같은 큐가 아니기 때문에 대기 상태가 발생하지 않습니다.
- **큐 구조 분리**: 메인 큐는 UI를 작업하고, 백그라운드는 데이터를 처리하며 역할을 분리합니다.

# 7️⃣ 메모리 구조

<img width="282" height="560" alt="image" src="https://github.com/user-attachments/assets/a9294680-8593-4e24-83f4-c7edeb3a5f40" />

프로그램이 실행되면 운영체제는 메모리(RAM)에 해당 프로그램을 위한 공간을 할당합니다.

이 메모리 공간은 일반적으로 다음과 같은 네 가지 영역으로 나뉘게 됩니다.

- Code
- Data
- Heap
- Stack

## Code 영역

> 실행할 코드가 저장되는 영역
> 

우리가 작성한 소스 코드는 컴파일 과정을 거쳐 기계어로 변환되며, 이 Code 영역에 저장됩니다.

이 영역의 특징은 다음과 같다.

<aside>

- 컴파일 시점에 결정됨
- 실행 중 변경되지 않음
- Read-Only 속성
</aside>

즉, 프로그램 실행 중에 코드가 변경되지 않도록 보호된 영역입니다.

## Data 영역

> 전역 변수와 static 변수가 저장되는 영역
> 

Data 영역에는 프로그램 전체에서 접근 가능한 전역 변수(global)와 static 변수가 저장됩니다.

<aside>

- 프로그램 시작 시 메모리에 할당됨
- 프로그램 종료 시 해제됨
- 실행 중 값 변경 가능 (Read-Write)
</aside>

일반적인 언어에서는 static 변수가 프로그램 시작과 동시에 메모리에 올라가지만,

Swift에서는 `static`이 **lazy하게 동작합니다.**

즉, 선언 시점이 아니라 **처음 접근하는 시점에 메모리에 할당됩니다.**

```swift
struct Example {
    static var value = {
        print("초기화")
        return 10
    }()
}

print("프로그램 시작")
// 아직 초기화 안됨

print(Example.value)
// 이 시점에서 "초기화" 출력
```

## Heap 영역

> 런타임에 동적으로 할당되는 메모리 영역
> 

Heap 영역은 실행 중에 크기가 결정되는 데이터를 저장하는 공간입니다.

주로 참조 타입(reference type)이 이 영역에 저장됩니다.

일반적인 언어에서는 `malloc`, `free` 등을 통해 직접 메모리를 관리하지만, Swift에서는 ARC가 있기 때문에 이러한 과정을 직접 다루지 않습니다.

다음과 같은 것들은 모두 Heap에 저장됩니다.

<aside>

- 클래스 인스턴스
- 클로저
</aside>

## Stack 영역

> 함수 호출과 지역 변수가 저장되는 영역
> 

Stack 영역은 함수 실행과 관련된 데이터가 저장되는 공간이다.

<aside>

- 함수 호출 시 생성
- 함수 종료 시 자동 제거
- 매우 빠른 접근 속도
</aside>

Stack에는 다음과 같은 것들이 저장됩니다.

<aside>

- 함수의 지역 변수
- 매개변수
- 함수 호출 정보
</aside>

# 8️⃣ 가상 메모리 / 캐시 메모리

## 가상 메모리

> 실제 물리 메모리(RAM)를 효율적으로 사용하기 위해, 더 큰 메모리 공간처럼 동작하도록 만든 개념
> 

앱은 실제 RAM보다 훨씬 큰 주소 공간을 사용하는 것처럼 보이지만, 운영체제는 이를 페이지 단위로 나누어 RAM과 디스크를 오가며 필요한 데이터만 RAM에 올려 관리합니다.

### iOS에서의 가상 메모리

iOS에서는 일반적인 데스크톱 OS처럼 디스크 스왑을 적극적으로 사용하지 않고 다음과 같은 방식을 사용합니다.

- **메모리 압축 (Compressed Memory):** 사용하지 않는 메모리 페이지를 **압축**하여 RAM에 저장함으로써 처리 속도를 유지하고 성능 저하를 방지합니다.
- 메모리 부족 시 앱 종료: iOS는 스왑보다 앱을 종료하는 전략을 사용

### 페이지 폴트

> 필요한 데이터가 RAM에 없어서 가져오는 과정
> 

앱이 필요한 데이터가 실제 메모리에 없을 때 디스크에서 RAM으로 가져오는 현상으로, 너무 자주 발생하면 성능이 저하됩니다.

## 캐시 메모리

> 자주 사용하는 데이터를 빠르게 접근하기 위해 임시로 저장하는 공간
> 

캐시는 운영체제가 아니라 **앱 레벨에서 성능 최적화를 위해 사용됩니다.**

- **NSCache:** Swift/Obj-C에서 제공하는 캐시 클래스로, 메모리 부족 상황(`didReceiveMemoryWarning`)에서 자동으로 데이터를 정리하여 앱이 종료되지 않도록 관리합니다.
- **디스크 캐시 (Disk Cache):** 앱의 데이터나 이미지를 디스크에 저장하는 방식으로, 앱 종료 후에도 데이터가 유지되며 `FileManager` 등을 통해 구현합니다.

## 메모리가 부족할 때는 어떻게 대응할까?

iOS는 메모리가 부족해지면 다음과 같은 순서로 대응하게 됩니다.

1. 불필요한 캐시 제거 (NSCache)
2. 메모리 압축
3. 백그라운드 앱 종료
4. 그래도 부족하면 현재 앱 종료

## 메모리 최적화 기법

### 이미지 메모리 최적화

- Lazy Loading
- 이미지 크기 최적화
- 캐싱

### 메모리 관리 기법

Swift는 ARC를 사용하여 메모리를 자동으로 관리합니다.

하지만 **순환 참조** 문제는 여전히 존재합니다.

순환 참조 문제를 해결하기 위해서는 weak self로 약한 참조를 사용하거나, unowned와 같은 비소유 참조를 사용합니다.

# 9️⃣ ARC / retain cycle / iOS 연결

## ARC

> 객체의 참조 횟수를 자동으로 관리하여, 더 이상 사용되지 않는 메모리를 해제하는 방식
> 

Swift에서는 메모리를 직접 해제하지 않고, ARC를 통해 자동으로 관리합니다.

ARC는 객체를 참조하는 개수(Reference Count)를 추적하다가,
그 값이 0이 되면 해당 객체를 메모리에서 해제합니다.

### ARC 동작 방식

<aside>

- 객체 생성 → Reference Count = 1
- 참조 증가 → +1
- 참조 해제 → -1
- Reference Count = 0 → 메모리 해제
</aside>

```swift
class Person {
    let name: String
    
    init(name: String) {
        self.name = name
        print("\(name) 생성")
    }
    
    deinit {
        print("\(name) 해제")
    }
}

var person: Person? = Person(name: "철수")
person = nil  // 이 시점에 메모리 해제
```

- `init` 시점에 RC +1
- `nil`로 만들면 **참조가 끊기면서 RC -1**
- RC가 0이 되면 → 메모리 해제

## Retain cycle (순환 참조)

> 서로가 서로를 강하게 참조하여, Reference Count가 0이 되지 않아 메모리가 해제되지 않는 현상
> 

```swift
class A {
    var b: B?
}

class B {
    var a: A?
}

var a: A? = A()
var b: B? = B()

a?.b = b
b?.a = a

a = nil
b = nil
```

이 코드에서는 A → B 참조, B → A 참조하게 되면서 서로를 참조하고 있기 때문에 둘 다 메모리가 해제되지 않습니다. 이 상황에서는 메모리 누수가 발생하게 됩니다.

이 메모리 누수를 해결하기 위해서는 참조를 약하게 만들어야 합니다.

### weak

> 참조 대상이 사라지면 자동으로 nil이 되는 참조
> 

```swift
class B {
    weak var a: A?
}
```

- 옵셔널 타입
- 안전함
- 가장 많이 사용

### unowned

> 참조 대상이 항상 존재한다고 가정하는 참조
> 

```swift
class B {
    unowned var a: A
}
```

- 옵셔널 아님
- 더 빠름
- 하지만 잘못 쓰면 crash
