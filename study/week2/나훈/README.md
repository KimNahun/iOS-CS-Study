## 1. 프로세스와 스레드

**프로세스 (Process)**

- 실행 중인 프로그램의 독립적인 인스턴스
- OS로부터 독립된 메모리 공간(Code, Data, Heap, Stack)을 할당받음
- 프로세스끼리는 메모리를 공유하지 않음 → 다른 프로세스가 죽어도 내 프로세스는 영향 없음
- 프로세스 간 통신이 필요하면 IPC(Inter-Process Communication) 사용

**스레드 (Thread)**

- 프로세스 내의 실행 단위
- 하나의 프로세스 안에 여러 스레드가 존재할 수 있음
- Stack만 개별적으로 가지고, Code / Data / Heap은 같은 프로세스의 스레드끼리 공유
- 공유 메모리 덕분에 통신이 쉽고 생성 비용이 낮음
- 대신 하나의 스레드가 잘못되면 같은 프로세스 내 다른 스레드도 영향을 받을 수 있음

| 구분 | 프로세스 | 스레드 |
|------|---------|--------|
| 메모리 | 독립된 공간 (Code/Data/Heap/Stack 모두 별도) | Stack만 별도, 나머지 공유 |
| 생성 비용 | 무거움 | 가벼움 |
| 통신 | IPC 필요 | 공유 메모리로 간단 |
| 안정성 | 다른 프로세스에 영향 없음 | 같은 프로세스 내 영향 있음 |

> **iOS 연결 포인트**
>
> - iOS 앱 하나 = 하나의 프로세스
> - 앱 안에서 여러 스레드가 동작 (메인 스레드 + 백그라운드 스레드들)
> - 메인 스레드는 UI 렌더링 전담, 백그라운드 스레드는 네트워크 / 이미지 처리 등 담당
> - 백그라운드에서 UI를 건드리면 크래시 나는 이유: 메인 스레드만 UIKit에 안전하게 접근 가능

## 2. 컨텍스트 스위칭

- CPU가 현재 실행 중인 프로세스/스레드를 멈추고 다른 것을 실행하도록 전환하는 과정
- 전환 전 현재 상태(레지스터, PC, 스택 포인터 등)를 PCB(Process Control Block)에 저장
- 전환 후 다음 프로세스의 저장된 상태를 복원해서 이어서 실행

**오버헤드**

- 컨텍스트 스위칭 자체는 CPU가 실제 작업을 하지 않는 낭비 시간
- 너무 자주 발생하면 오버헤드가 커져서 전체 성능 저하
- 스레드의 컨텍스트 스위칭은 프로세스보다 비용이 낮음 (공유 메모리가 많아서 저장/복원할 게 적음)

**언제 발생하나?**

- I/O 작업으로 대기 상태가 될 때
- 타임 슬라이스(할당된 CPU 시간)가 끝났을 때
- 우선순위 높은 프로세스가 들어왔을 때

> **iOS 연결 포인트**
>
> - GCD가 내부적으로 스레드 풀(Thread Pool)을 관리하면서 컨텍스트 스위칭 비용을 최소화
> - `DispatchQueue.global()`에 너무 많은 작업을 던지면 스레드 폭발(Thread Explosion) → 컨텍스트 스위칭 과부하
> - Swift Concurrency(async/await)의 등장 이유 중 하나도 이 스레드 낭비 문제를 줄이기 위함

## 3. 동기와 비동기

**동기 (Synchronous)**

- 작업이 끝날 때까지 현재 스레드가 기다림
- 다음 코드는 이전 작업이 완전히 끝나야 실행됨
- 순서가 보장되지만 기다리는 동안 스레드가 블로킹됨

**비동기 (Asynchronous)**

- 작업을 던져놓고 현재 스레드는 바로 다음으로 진행
- 작업이 완료되면 콜백/클로저/completion handler로 결과를 받음
- 스레드를 블로킹하지 않아서 효율적

| 구분 | 동기 | 비동기 |
|------|------|--------|
| 실행 흐름 | 순차적 (끝날 때까지 대기) | 비순차적 (결과는 나중에) |
| 스레드 | 블로킹 | 논블로킹 |
| 사용 예 | 간단한 연산, 순서가 중요한 작업 | 네트워크, 파일 I/O |

> **iOS 연결 포인트**
>
> - 네트워크 요청은 비동기로 처리 → 메인 스레드가 블로킹되면 UI가 멈춤
> - URLSession의 completion handler가 비동기 패턴의 대표 예
> - `DispatchQueue.main.async { }` → 비동기로 메인 큐에 UI 업데이트 작업을 던짐
> - async/await 도입으로 비동기 코드를 동기처럼 읽기 쉽게 작성 가능

## 4. 직렬과 병렬

**직렬 (Serial)**

- 작업을 하나씩 순서대로 처리
- 이전 작업이 끝나야 다음 작업이 시작됨
- 순서가 보장되고 데이터 경쟁이 없음

**병렬 (Parallel)**

- 여러 작업을 동시에 처리
- 멀티 코어 CPU에서 진짜로 동시에 실행됨
- 처리량이 높지만 순서 보장이 안 되고 Race Condition 위험

**동시성(Concurrency) vs 병렬성(Parallelism)**

- 동시성: 여러 작업이 동시에 실행되는 것처럼 보이는 것 (싱글 코어에서도 가능, 빠르게 번갈아 실행)
- 병렬성: 실제로 여러 작업이 동시에 실행되는 것 (멀티 코어 필요)
- 동시성이 더 넓은 개념, 병렬성은 동시성의 한 형태

| 구분 | 직렬 | 병렬 |
|------|------|------|
| 실행 방식 | 순차 처리 | 동시 처리 |
| 순서 보장 | O | X |
| 속도 | 상대적으로 느림 | 빠름 |
| 안전성 | Race Condition 없음 | Race Condition 주의 |

> **iOS 연결 포인트**
>
> - `DispatchQueue(label: "...", attributes: .concurrent)` → 병렬 큐
> - `DispatchQueue(label: "...")` (기본값) → 직렬 큐
> - 메인 큐는 직렬 큐 → UI 업데이트가 순서대로 처리됨
> - 이미지 다운로드 같은 독립적인 작업은 concurrent 큐로 성능 향상
> - Swift actor는 내부적으로 직렬 실행을 보장해서 데이터 경쟁 방지

## 5. Race Condition

- 두 개 이상의 스레드가 공유 데이터에 동시에 접근하고, 최소 하나가 쓰기 작업을 할 때 발생
- 실행 순서에 따라 결과가 달라지는 예측 불가능한 버그
- 재현이 어렵고 찾기가 까다로운 문제

**예시**

```
count = 0
스레드 A: count 읽음 (0) → +1 계산 → count에 1 씀
스레드 B: count 읽음 (0) → +1 계산 → count에 1 씀
결과: count = 1 (기대값은 2)
```

→ A와 B가 동시에 읽어버려서 서로의 쓰기가 덮어씌워짐

**해결 방법**

- **Mutex (Mutual Exclusion)**: 한 번에 하나의 스레드만 접근 허용, 나머지는 대기
- **Semaphore**: 동시에 접근 가능한 스레드 수를 제한 (Mutex는 1개 제한, Semaphore는 N개 제한)
- **Lock**: 임계 구역(Critical Section)에 진입할 때 잠금, 나올 때 해제

> **iOS 연결 포인트**
>
> - GCD의 `DispatchBarrier`: 읽기는 동시에 허용, 쓰기만 단독 실행 → 읽기 성능 유지하면서 Race Condition 방지
> - `NSLock`, `os_unfair_lock`: Swift에서 Mutex 구현
> - Swift actor: 내부 상태를 직렬로 보호해서 Race Condition 원천 차단
> - UserDefaults 멀티스레드 동시 접근도 Race Condition 위험 → 주의 필요

## 6. Deadlock

- 두 개 이상의 스레드가 서로 상대방이 가진 자원을 기다리면서 영원히 진행되지 못하는 상태

**발생 조건 (4가지 모두 충족 시 발생)**

1. **상호 배제 (Mutual Exclusion)**: 자원을 동시에 사용 불가
2. **점유와 대기 (Hold and Wait)**: 자원을 가진 채로 다른 자원을 기다림
3. **비선점 (No Preemption)**: 강제로 자원을 빼앗을 수 없음
4. **순환 대기 (Circular Wait)**: A → B → A 형태로 순환 의존

**예시**

```
스레드 A: Lock1 보유 → Lock2 기다림
스레드 B: Lock2 보유 → Lock1 기다림
→ 둘 다 영원히 대기
```

**해결/예방 방법**

- 잠금 순서 통일: 모든 스레드가 같은 순서로 Lock 획득
- 타임아웃 설정: 일정 시간 기다려도 안 되면 포기하고 재시도
- 순환 대기 조건 제거

> **iOS 연결 포인트**
>
> - `DispatchQueue.main.sync { }` 를 메인 스레드에서 호출하면 데드락 발생
>   - 메인 스레드가 자기 자신이 끝나길 기다리는 순환 구조
> - 직렬 큐에서 같은 큐에 `sync`로 작업을 던지면 동일한 문제 발생
> - 실무에서 데드락은 주로 `sync` 남발에서 발생 → `async` 기본 원칙

## 7. 메모리 구조 (코드, 데이터, 힙, 스택)

프로세스가 실행되면 OS는 메모리를 아래 4개 영역으로 나눠서 할당함

| 영역 | 내용 | 특징 |
|------|------|------|
| Code (Text) | 실행할 프로그램 코드 | 읽기 전용, 컴파일 시 크기 결정 |
| Data | 전역 변수, static 변수 | 프로그램 시작 시 할당, 종료 시 해제 |
| Heap | 동적 할당 메모리 | 런타임에 크기 결정, 낮은 주소 → 높은 주소 방향으로 성장 |
| Stack | 함수 호출, 지역 변수, 리턴 주소 | 컴파일 시 크기 결정, 높은 주소 → 낮은 주소 방향으로 성장 |

**스택 vs 힙 비교**

| 구분 | Stack | Heap |
|------|-------|------|
| 할당 시점 | 컴파일 타임 | 런타임 |
| 관리 주체 | OS (자동) | 개발자 / ARC (수동 또는 반자동) |
| 속도 | 빠름 | 상대적으로 느림 |
| 크기 | 제한적 | 상대적으로 큼 |
| 데이터 | 지역 변수, 함수 프레임 | 동적 할당 객체 |

**스택 오버플로우**

- Stack이 너무 커져서 Heap 영역과 충돌할 때 발생
- 재귀 함수가 탈출 조건 없이 무한 호출되면 발생

> **iOS 연결 포인트**
>
> - Swift **값 타입** (struct, enum, tuple) → **Stack** 에 저장
> - Swift **참조 타입** (class, closure) → **Heap** 에 저장
> - Stack 할당이 Heap보다 빠름 → 성능 민감한 곳에서 struct를 선호하는 이유
> - 클로저가 외부 변수를 캡처하면 해당 변수가 Heap으로 이동 (캡처 리스트 주의)
> - 메인 스레드와 백그라운드 스레드는 각자 별도의 Stack을 가짐

## 8. 가상 메모리 / 캐시 메모리 기초

**가상 메모리 (Virtual Memory)**

- 실제 물리 메모리(RAM)보다 더 큰 주소 공간을 프로세스에 제공하는 기술
- 프로세스는 자신이 물리 메모리 전체를 독점하는 것처럼 동작
- 실제 물리 주소로의 변환은 OS와 MMU(Memory Management Unit)가 담당
- 자주 쓰는 데이터만 RAM에, 나머지는 디스크(스왑 영역)에 저장

**페이지(Page)와 페이지 폴트(Page Fault)**

- 가상 메모리는 일정 크기의 **페이지** 단위로 관리
- 프로세스가 접근한 페이지가 RAM에 없으면 → **페이지 폴트** 발생
- OS가 디스크에서 해당 페이지를 RAM으로 불러옴 → 비용 큼

**캐시 메모리 (Cache Memory)**

- CPU와 RAM 사이의 속도 차이를 줄이기 위한 초고속 메모리
- L1 → L2 → L3 → RAM 순서로 계층 구성 (L1이 가장 빠르고 작음)
- **캐시 히트**: 필요한 데이터가 캐시에 있음 → 빠름
- **캐시 미스**: 필요한 데이터가 캐시에 없음 → RAM에서 가져옴 → 느림

**지역성 원리 (Locality)**

- **시간적 지역성**: 최근에 접근한 데이터는 곧 다시 접근할 가능성이 높음
- **공간적 지역성**: 접근한 데이터 주변 데이터도 곧 접근할 가능성이 높음
- 배열이 연결 리스트보다 실제로 빠른 이유 → 배열은 연속 메모리라 캐시 효율이 높음

> **iOS 연결 포인트**
>
> - iOS는 스왑(Swap) 파티션이 없음 → 메모리 부족 시 OS가 앱을 강제 종료 (메모리 경고 → 앱 종료)
> - `applicationDidReceiveMemoryWarning` / `didReceiveMemoryWarning` 으로 메모리 경고 감지
> - 이미지 캐시 라이브러리(SDWebImage, Kingfisher)가 필요한 이유: 같은 이미지를 반복 요청하지 않도록 메모리/디스크 캐시 활용
> - Swift Array가 연결 리스트보다 빠른 이유도 캐시 지역성 덕분

## 9. ARC / Retain Cycle / iOS 연결

**ARC (Automatic Reference Counting)**

- Swift가 Heap 메모리를 자동으로 관리하는 방식
- 참조 타입(class, closure) 객체의 **참조 카운트(Reference Count)** 를 추적
- 참조 카운트가 0이 되면 메모리에서 자동 해제
- 컴파일 타임에 retain/release 코드를 자동으로 삽입해줌 (GC와 달리 런타임 오버헤드 없음)

**참조의 종류**

| 참조 종류 | 참조 카운트 증가 | nil 가능 | 설명 |
|----------|----------------|---------|------|
| 강한 참조 (strong) | O | O | 기본값, 카운트 증가 |
| 약한 참조 (weak) | X | O (Optional) | 카운트 증가 없음, 대상 해제 시 자동 nil |
| 미소유 참조 (unowned) | X | X (non-Optional) | 카운트 증가 없음, 대상 해제 시 크래시 위험 |

**Retain Cycle (강한 참조 순환)**

- 두 객체가 서로를 강하게 참조하면 참조 카운트가 절대로 0이 안 됨 → 메모리 누수
- 예: A가 B를 strong 참조, B가 A를 strong 참조 → 둘 다 해제 불가

```
A → (strong) → B
B → (strong) → A
둘 다 카운트 1 유지 → 영원히 해제 안 됨
```

**해결 방법**

- 한쪽을 `weak` 또는 `unowned`로 선언해서 순환 끊기
- 클로저가 self를 캡처할 때 `[weak self]` 또는 `[unowned self]` 캡처 리스트 사용

> **iOS 연결 포인트**
>
> - delegate 패턴에서 `weak var delegate`가 표준인 이유: ViewController가 delegate를 strong으로 들고, delegate가 VC를 strong으로 들면 Retain Cycle 발생
> - 클로저에서 `[weak self]` 캡처 리스트: 클로저가 self를 strong으로 캡처하면 VC가 pop되도 해제 안 됨
> - Instruments의 **Leaks** / **Allocations** 도구로 Retain Cycle 탐지 가능
> - `deinit`이 호출되지 않는다면 Retain Cycle 의심
