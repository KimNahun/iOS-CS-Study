## 프로세스와 스레드

- 프로그램
    - 컴퓨터가 실행할 수 있는 명령어, 코드 덩어리
- 프로세스
    - 프로그램을 메모리에 적재하고, CPU 자원을 할당받아서 실행하는 것
    - 특징
        - 각 프로세스는 OS로부터 독립된 메모리 공간을 할당받는다
    - 단일 프로세스
        - 초기 시스템은 단일 프로세스였다.
            - 한번에 하나의 프로그램만 실행할 수 있다.
            - 다른 프로그램을 실행하려면, 실행중인 프로그램을 종료해야한다.
        - CPU 사용률이 좋지 않다.
            - CPU가 놀고 있는 시간이 많다. (특히 I/O 작업을 할 경우)
    - 멀티 태스킹
        - 여러 개의 프로그램을 동시에 실행할 수 있게 하는 초기 방법
            - 여러 개의 프로그램을 메모리에 동시에 올려두고, CPU를 번갈아 가면서 사용한다.
            - 여러 프로세스가 동시에 실행되는 것 처럼 보인다.
        - 단점
            - 하나의 프로세스 자체는 여전히 동시에 하나의 작업밖에 수행할 수 없다.
            - **문맥 교환 (Context Switching)**이 발생한다. 매우 무거운 작업!
            - 각 프로세스가 분리된 메모리 공간을 할당받아 사용하기 때문에, 프로세스 간 데이터 공유가 까다롭다.
    - 멀티 프로세싱
        - CPU의 코어가 여러개로 발전함
    - 용어 정리
        - 멀티태스킹 : 하나의 코어를 두고 2개 이상의 스레드가 경합하는가?
        - 멀티스레딩 : 하나의 프로세스가 두개 이상의 스레드?
        - 멀티프로세싱 : 2개 이상의 코어?
    - 예제
        - 싱글코어 CPU, 싱글스레드 프로세스 2개 >> 멀티태스킹O, 멀티스레딩X, 멀티프로세싱X
        - 싱글코어 CPU, 듀얼스레드 프로세스 1개 >> 멀티태스킹O, 멀티스레딩O, 멀티프로세싱X
        - 듀얼코어 CPU, 싱글스레드 프로세스 2개 >> 멀티태스킹X, 멀티스레딩X, 멀티프로세싱O
        - 듀얼코어 CPU, 듀얼스레드 프로세스 1개 >> 멀티태스킹X, 멀티스레딩O, 멀티프로세싱O
        - 듀얼코어 CPU, 듀얼스레드 프로세스 2개 >> 멀티태스킹O, 멀티스레딩O, 멀티프로세싱O
- 스레드
    - 스레드는 프로세스 내에서의 실행 흐름에 대한 개념이다.
        - 각 프로세스는 하나 이상의 스레드를 가질 수 있다.
        - CPU에서 실행되는 단위이다.
        - 어떤 프로세스의 스레드들은 메모리를 공유한다. → 문맥 교환이 가볍다!

## 컨텍스트 스위칭

- 컨텍스트 스위칭이란?
    - 멀티태스킹에서 n개의 프로세스가 동일한 프로세서를 두고 경합한다.
    - n개의 프로세스가 동일한 프로세서를 번갈아가면서 사용하는 것이, 동시에 실행되는 것 처럼 보인다
    - 동일한 프로세서가 n개의 프로세스(스레드)를 번갈아가면서 실행하기 위해, 실행중이던 프로세스(스레드)를 교체하는 것을, 컨텍스트 스위칭이라고 한다.
- 컨텍스트 스위칭은 언제 일어날까?
    1. 경합중인 프로세스가 한번에 실행할 수 있는 최대시간(Time Slice, Time Quantim)을 다 사용한 경우
    2. I/O 작업을 하는 경우
    3. 다른 리소스를 기다리는 경우
    4. 인터럽트가 발생한 경우
- 컨텍스트 스위칭의 주체
    - OS의 커널(kernel)에 의해 수행된다.
- 컨텍스트 스위칭의 과정
    1. 동일 프로세스의 스레드 간 스위칭 (스레드 컨텍스트 스위칭)
        - 커널 모드에서 실행된다. 통제권이 커널에게 있다.
        - CPU의 레지스터 상태를 교체한다.
    2. 다른 프로세스의 스레드 간 스위칭 (프로세스 컨텍스트 스위칭)
        - 추가로, 가상 메모리 주소 관련 처리를 수행한다.
            - MMU(Memory Management Unit)
                - 가상 메모리와 물리 메모리 사이의 주소 변환을 담당한다.
                - 메모리 주소를 추상화해서 각 프로세스가 독립적인 주소 공간을 갖는 것 처럼 만든다.
            - TLB(Translation Lookaside Buffer)
                - MMU 내에 존재하는 캐시 메모리.
        - 동일 프로세스의 스레드 간 스위칭의 경우, 교체되는 스레드간에 메모리를 공유하므로 가상 메모리 주소 관련 처리 불필요
- 컨텍스트 스위칭의 문제점 - 캐시 오염
    - 또 사용할 것으로 예상하고 캐시에 저장해둔 값이 의미가 없는 데이터가 되어버린다..

## 동기와 비동기

- 동기
    - 일이 들어온 순서대로, 순차적으로 처리한다.
    - 먼저 시작한 작업이 끝날 때 까지, 다음 작업은 중단 및 대기
- 비동기
    - 작업 시간이 오래 걸리거나, 언제 끝날지 예측하기 힘든 작업을 **메인스레드**에서 **다른 스레드**로 전달하여 처리한다. (ex. API 요청)
    - 다음 작업을 시작하려면 이전 작업이 끝날 때 까지 기다려야하는 동기와 달리, 이전 작업이 다른 스레드에서 처리되는 동안 **다음 작업을 바로 진행**할 수 있다
    - 다른 스레드에서 **이전 작업이 완료**되고 나면 **메인스레드에서 이어서 작업**한다.

## 직렬과 병렬

- 직렬
    - 작업을 하나의 스레드에서 순차적으로 처리하는 것
- 병렬
    - **작업**을 **여러 작업**으로 쪼개서 여러 스레드에 분산시켜서 동시에 처리하는 것
    - **독립적이지만 유사한 작업**일 경우 병렬 처리가 유용하다

    - 메인 스레드에서 수행해야하는 작업을 어떻게 처리할지 결정하는 것
    - 메인 스레드에서 작업을 다른 스레드에게 전달해서 시키고, 메인 스레드는 다음 작업을 바로 진행한다.
        - 비동기 작업을 호출한 메인스레드가 비동기 작업의 완료를 기다린다!

        - I/O 작업 (오래 걸린다)
- 병렬
    - 멀티코어 프로세싱

- 동기 + 직렬
    - 가장 단순하다
    - 하나의 스레드가 작업을 순차적으로 처리
- 비동기 + 직렬
    - 이전 작업은 다른 스레드에게 시키고, 다음 작업을 바로 진행한다
- 동기 + 병렬
    - 하나의 작업을 여러 작업으로 쪼개서 동시에 처리함 + 다음 작업은 바로 진행하지 않고 대기
- 비동기 + 병렬
    - 하나의 작업을 여러 작업으로 쪼개서 다른 스레드들에서 동시에 처리함 + 다음 작업을 바로 진행하기
    - 다른 스레드들에서 동시에 처리하던 이전 작업이 완료되고 나면, 이어서 작업한다

## Race Condition, Deadlock

- 경쟁조건 (Race Condition)
    - 여러 프로세스/스레드가 동시에 같은 데이터를 조작할 때, 타이밍/접근 순서에 따라 결과가 달라질 수 있다.
    - 동일한 데이터를 조작하는 작업A, 작업B가 여러 프로세스/스레드에서 동시에 작업을 수행할 경우, 데이터의 결과를 예측할 수 없다
- 동기화 (Synchronization)
    - 여러 프로세스/스레드가 동시에 실행되어도, 공유 데이터의 일관성을 유지하는 것
- 임계 영역 (Critical Section)
    - 공유 데이터의 일관성을 보장하기 위해, 하나의 프로세스/스레드만 진입하여 실행할 수 있도록 해야하는 영역
    - 동일한 데이터를 조작하는 작업A, 작업B가 여러 프로세스/스레드에서 동시에 작업을 수행할 경우, 데이터의 결과를 예측할 수 없다
        - 작업A, 작업B 중 하나의 작업이 먼저 데이터에 접근할 경우, 해당 데이터 영역은 임계 영역이다.
        - 다른 작업은 이전 작업이 종료되어, 데이터 영역이 더이상 임계 영역이 아니게 될 때까지 기다리도록 한다.
            
            → 데이터의 결과를 예측할 수 있다!
            

---

- Deadlock
    - 임계 영역에 대한 관리가 Thread Safe 하지 않을 때 발생하는 증상 중 하나
    - 자원을 점유하고 있고 + 동시에 다른 자원을 요청하며, 무한히 기다리는 상태
        - A는 1을 점유하고 2를 요청하며 대기
        - B는 2를 점유하고 1을 요청하며 대기
        - A와 B는 무한히 대기하고, 자원 1, 2도 무한히 점유상태에서 빠져나오지 못한다
- 상호 배제 (Mutual Exclusion)
    - Thread Safe를 지키기 위한 방법 중 하나.
    - 락(Lock)을 통해서 달성한다.
        1. Spinlock
            
            ```swift
            while(testAndSet(&lock) == 1) {} // 결과가 1이면 사용중임
            ```
            
            - 접근 권한을 획득할 때 까지 반복해서 확인하는 방법
            - CPU 낭비가 발생할 수 있다.
                - 임계 영역에서 실행되는 작업이 문맥 교환보다 더 빠르다면, Mutex보다 효율적이다.
        2. Mutex
            - 접근 권한을 확인해보고, 실패하면 대기열에 들어가서 잠든다.
            - 이전 작업이 임계 영역에서 빠져나오면, OS가 대기열에 있던 스레드를 깨운다.
            - NSLock / NSRecursiveLock
                
                ```swift
                lock.lock()
                defer {
                	lock.unlock()
                }
                
                /* do something */
                ```
                
                - Mutex 방법임
            - NSLock과 NSRecursiveLock 차이점
                - NSLock
                    - 하나 이상의 스레드에 대한 상호배제 방법
                    - A 스레드가 lock을 걸었을 때, unlock할 때까지 A 스레드를 제외한 모든 스레드가 접근할 수 없다.
                    - A 스레드가 lock을 걸었을 때, A 스레드가 다시 접근하여 lock걸 수 없다.
                        
                        → A 스레드가 A 스레드의 unlock을 대기한다. **데드락 발생 위험**
                        
                - NSRecursiveLock
                    - 둘 이상의 스레드에 대한 상호배제 방법
                    - A 스레드가 lock을 걸었을 때, unlock할 때까지 다른 스레드는 접근할 수 없다.
                    - A 스레드가 lock을 걸었을 때, A 스레드가 다시 접근하여 lock걸 수 있다. (lock 횟수만큼 unlock해야한다)
        3. Semaphore
            - n개의 프로세스/스레드가 임계영역에 동시에 접근할 수 있도록 제한할 수 있는 방법
            - DispatchSemaphore
                
                ```swift
                let semaphore = DispatchSemaphore(value: 1)
                DispatchQeueue.global().async {
                	semaphore.wait() // 임계영역 설정
                	let book = books.removeFirst()
                	print(book)
                	semaphore.signal() // 임계영역 해제
                }
                ```
                

## 메모리 구조: 코드, 데이터, 힙, 스택

- **코드 영역**
    - 실행할 프로그램의 코드를 저장하는 영역이다. 코드 영역에 저장된 코드를 CPU가 읽어서 실행한다.
    - 프로그램의 시작에 로드되고, 프로그램의 종료까지 남아있는다.
    - **텍스트 영역**이라고도 한다
- 데이터
    - 프로그램의 전역 변수, 정적 변수가 저장되는 영역이다.
    - 프로그램의 시작에 로드되고, 프로그램의 종료까지 남아있는다
- 힙
    - 사용자에 의해 런타임에 직접 공간을 할당, 해제하는 영역
        - C언어 등에서는 new, malloc을 통해 메모리를 할당하고 free, delete를 통해 메모리를 해제한다.
        - Java는 GC를 통해 메모리를 자동으로 관리한다.
        - Swift는 ARC를 통해 메모리를 할당하고 해제한다.
    - 힙 영역의 크기는 런타임에 크기가 결정된다
    - FIFO
        - 불연속적인 메모리 공간에 위치
- 스택
    - 함수 호출시 생성되는 지역 변수, 매개 변수를 저장하는 영역이다.
    - 함수의 호출과 함께 할당되며, 함수가 종료되면 함께 소멸한다.
    - 스택 프레임
        - 스택에 저장되는 함수의 호출정보
        - 함수가 종료되며 스택이 소멸하고 이전 상태로 돌아갈 때 사용된다.

    - 스택 영역의 크기
        - 각 메서드가 호출되었을 때 차지하는 스택 프레임의 크기는 컴파일 타임에 결정된다.
        - 스택 영역의 전체 크기는 런타임에 결정된다.
    - LIFO
        - 연속적인 메모리 공간에 위치
- 힙 오버플로우 / 스택 오버플로우

    - 오버플로우가 발생할 경우, 에러와 함께 강제 종료된다.

- 가상 메모리란?
    - 주기억장치(RAM)의 공간이 부족할 때 부기억장치(SSD, HDD)를 주기억장치처럼 사용할 수 있도록 하는 기술
    - 장점
        - 메모리의 물리적 크기가 부족해도 프로그램이 중단되지 않고 계속 실행될 수 있도록 돕는다.
        - 각 프로세스에 독립적인 가상 메모리 공간을 할당함으로서, 다른 프로세스의 메모리 영역을 침범하지 못하게 한다.
        - 프로세스는 할당 받은 메모리 공간 전체를 항상 사용하고 있지 않으므로, 실제로 필요한 부분만 적재하며 한정적인 자원을 효율적으로 사용할 수 있다.
    - 단점
        - 페이지 부재 비용 : 부기억장치에서 주기억장치로 데이터를 불러오는 데에 시간이 걸린다
        - 복잡성 : 페이지 테이블 관리, 페이지 교체 관리로 인한 복잡성 증가
        - 성능 저하 : 페이지 부재시 부기억장치의 속도가 주기억장치보다 훨씬 느리므로 성능 저하가 발생한다
- 가상 메모리의 동작 원리
    1. 페이징 (Paging)
        - 원리
            - 가상 메모리와 물리 메모리를 같은 크기의 블록으로 나눈다.
            - 운영체제가 프로그램이 필요로하는 페이지를 메모리에 적재한다.
        - 용어
            - 페이지 : 가상 메모리를 나눈 블록. 크기가 일정하고, 프레임과 크기가 동일하다.
            - 프레임 : 물리 메모리를 나눈 블록. 크기가 일정하고, 페이지와 크기가 동일하다.
            - 페이지 테이블 : 프로세스가 알고 있는 가상 주소와, 메모리의 실제 주소를 매핑하는 테이블. 페이지 번호, 페이지 오프셋 등을 알고있다.
        - 동작 과정
            - 가상주소→물리주소 변환 : 프로그램이 데이터를 읽거나 쓸 때 가상 주소를 참조한다. MMU(Memory Management Unit)이 페이지 테이블을 참조하여 주소를 변환한다.
            - 페이지 부재 (page fault) : 프로그램이 접근하려는 페이지가 주기억장치가 없는 경우 **페이지 부재**가 발생했다고 말한다. 운영체제에 의해 부기억장치의 페이지를 주기억장치로 불러온다.
        - 내부 단편화
            - 하나의 페이지 안에 논리적으로 무관하여 사용되지 않는 공간이 발생할 수 있다.
    2. 세그멘테이션 (Segmentation)
        - 원리
            - 데이터 영역을 단순히 같은 크기로 나누지 않고, 논리적으로 관련 있는 여러개의 세그먼트로 나누어서 메모리에 할당한다.
        - 용어
            - 가상 주소 : 세그멘테이션에서의 가상 주소는 ‘세그먼트 번호 + 오프셋’으로 이루어져있다.
            - 세그먼트 테이블 : 세그먼트 번호와 오프셋, 세그먼트 번호에 대응되는 물리 주소, 세그먼트의 길이(한계값)을 알고 있다.
                - 길이(한계값) : 오프셋이 길이(한계값)을 초과할 경우 **잘못된 메모리 접근**이다.
        - 외부 단편화
            - 가변 크기로 인해, 하나의 세그먼트가 여러 조각으로 나뉘어 저장되거나, 각 세그먼트 사이에 버려지는 공간이 발생할 수 있다.
- 가상 메모리 정책
    
    페이지 부재(page fault)가 자주 발생할 경우 I/O의 느린 속도로 인해 속도 저하가 발생한다. 
    
    - 반입 정책 (Fetch Policy)
    - 교체 정책 (Replacement Policy)
        
        주기억장치가 꽉 찼을 경우, 새로운 페이지를 저장하기 위해 기존에 불러온 데이터를 주기억장치에서 지워야한다. 자주 사용하지 않는 페이지를 선별하여 교체하기 위하여 다음 기법들을 사용한다.
        
        - 최적 정책 : 미래에 참조될 시간이 가장 긴 페이지를 교체… (현실적으로 구현 불가능)
        - LRU (Least Recently Used) : 가장 오랫동안 참조되지 않은 페이지를 교체 (최고 성능, 오버헤드 발생)
        - FIFO : 가장 오래 전에 반입된 페이지를 교체
        - 클록 정책 (Clock Policy)
            - LRU의 오버헤드를 줄이면서 유사한 성능을 내는 알고리즘
            1. 주기억장치의 페이지를 참조할 때, 참조 비트를 1로 표시한다.
            2. 주기억장치의 메모리 공간을 연속된 원형으로 놓고 생각하며, 순차적으로 참조 비트를 확인한다.
                - 참조 비트가 0이면 한바퀴를 도는 동안 참조되지 않은 것으로 판단하고 주기억장치에서 내보낸다.
                - 참조 비트가 1이면 한바퀴를 도는 사이에 참조된 것이므로, 재사용 가능성이 있다고 판단하고 유지한다. 참조 비트는 0으로 변경한다.
                    - 다시 한바퀴를 돌아서 참조비트를 확인하게 되었을 때 1이면, 메모리에 적재된지 시간이 흘렀을에도 여전히 사용되고 있으므로 유지한다!
    - 페이지 버퍼링
        - 주기억장치에 있는 페이지/세그먼트를 지우기 전에, 부기억장치에 값을 저장해서 동기화해야한다
        - I/O는 속도가 느리므로, **변경 페이지 리스트**로 변경되었는지 여부를 관리하며 클러스터로 묶어서 한번에 입출력한다.

## 캐시 메모리 기초

<aside>
💡

CPU ↔ 캐시 ↔ 주기억장치

</aside>

- 캐시 메모리란?
    - CPU는 프로그램을 실행하기 위해 주기억장치에 적재된 데이터를 가져온다.
    - CPU의 저장공간은 매우 작고, 주기억장치는 CPU의 저장공간보다 속도가 매우 느리다. 따라서 매번 CPU와 주기억장치가 직접 데이터를 주고 받으면 성능이 저하된다.
    - CPU와 주기억장치 사이에, 주기억장치보다 빠른 캐시메모리를 두어 이를 보완한다.
- 캐시의 크기와 성능
    - 캐시의 전체 크기가 크면 Cache Hit가 증가한다
- 캐시의 인출 방식
    - 요구 인출 (Demand Fetch) : 필요시 인출
    - 선 인출 (Pre-Fetch) : 필요할 것으로 예상되는 데이터를 미리 인출
- 캐시의 쓰기 정책
    - Write-Through : CPU가 캐시에 저장된 값을 변경했을 때, 동시에 주기억장치의 값도 변경
    - Write-Back : 캐시 공간이 부족해서 값을 지울 때가 되어서야 주기억장치의 값을 변경
- 캐시의 교체 정책
    - Optimal
    - FIFO
    - LRU
    - LFU (Least Frequently Used)
- 공간 지역석 / 시간 지역성
    - (생략)
- 캐시 매핑 방법
    
    크기가 큰 주기억장치의 일부 데이터를 작은 크기의 캐시에 저장할 때, 캐시의 어느 공간에 저장할지 결정하는 방법
    
    - 직접 사상
        - 주기억장치에서의 메모리 주소를 캐시에서의 위치로 변환하여 결정한다.
        - 장점 : 구현하기 쉽다.
        - 단점 : 적중률이 낮다..
    - 연관 사상
        - 메모리 주소와 무관하게, 효율적인 교체 정책으로 위치를 선정하여 저장하는 방법
        - 장점 : 적중률이 높다
        - 단점 : 캐시에서 데이터를 찾을 때 전체 공간을 순회해야하므로 오버헤드가 발생한다.
    - 세트 연관 사상.
        - 직접 사상 + 연관사상
        - 메모리 주소를 기반으로 위치를 결정 & 결정된 위치가 n개
        - 장점
            - 메모리 주소로 캐시에서의 위치를 바로 알 수 있다.
            - 캐시 전체를 순회하지 않고, n개의 위치만 확인하면 된다.
            - 직접 사상보다 적중률이 높다

# iOS 연결 포인트

## 메인 스레드에서 UI를 처리하는 이유

- RunLoop
    
    ![image.png](attachment:63d5e700-1d04-4310-8059-68b2e9ba2842:image.png)
    
    - RunLoop
        - 스레드가 처리해야하는 작업이 있는지 모니터링한다.
        - 이를 통해 처리할 작업이 있을 때는 스레드를 바쁘게 유지하고, 작업이 없을 때는 스레드를 쉬도록 한다.
        - 각 스레드는 RunLoop를 갖는다.
    - 두가지 이벤트 수신
        - Input Source - 비동기 이벤트
            - 시스템 이벤트(터치, 마우스, 키보드 입력 등), 프로세스 간 통신을 모니터링한다.
            - 다른 스레드/앱으로부터 오는 비동기 이벤트를 모니터링한다.
        - Timer Source - 동기 이벤트
            - 예약된 시간에 또는 반복적으로 발생하는 타이머 이벤트를 모니터링한다.
        - RunLoop는 모니터링을 통해 이벤트를 감지하면, 이벤트 핸들러를 수행한다.
    - 루프 모드
        - 모니터링 하는 소스의 종류에 따라 여러가지 루프 모드가 있다.
    - 생성
        - 스레드의 RunLoop를 실행할 때 RunLoop가 존재하지 않으면 생성된다. (싱글톤)
    - 실행
        - 메인스레드의 RunLoop는 UIApplication에 의해 일정 주기마다 실행된다.
        - 백그라운드 스레드의 RunLoop는 필요시 직접 실행한다.
            
            > EX. 백그라운드 스레드에서 Timer를 설정할 경우, RunLoop를 직접 실행하지 않으면 작동하지 않는다.
            > 
- UIKit의 대부분의 타입은 Thread-Unsafe다.
    - Thread-safe하게 만들면 다음과 같은 것들을 고려해야한다
        - 여러 스레드에서 비동기적으로 View의 속성을 변경했을 때, 각 스레드의 runloop에 따라서 반영되어야 할까, 동시에 반영되어야할까?
        - 한 스레드에서는 UITableView의 특정 IndexPath를 삭제하고, 다른 스레드에서는 접근한다면, 충돌이 발생해야할까?
        - 한 스레드에서는 View를 삭제하고 있고, 다른 스레드에서는 View와의 상호작용을 시도한다면, 상호작용이 이뤄져야할까?
    - 위와 같은 많은 경우를 해결하기 위해 복잡도를 높이는 것 대신, Main-Thread에서 UI 작업을 실행할 것을 강제한다.
        - Main Thread의 RunLoop만이 화면 업데이트와 디스플레이와 동기화된다.
        - Global Thread에서의 작업은 화면 업데이트에 반영되지 않는다..
- View Drawing Cycle
    - View의 변경사항을 화면에 반영하는 과정
    - 과정
        1. View가 로드되면 시스템에 의해 View의 draw메서드가 호출된다.
        2. View를 그린 후 캡쳐하여 전달한다.
        3. View 변경시, 관련 메서드를 호출하여 시스템에 업데이트를 요청
        4. 다음 View Drawing Cycle에서 업데이트 요청받은 View가 업데이트된다.
        
        > View를 캡쳐하고 뿌리는 과정을 반복
        > 
    - 관련 메서드
        1. setNeedsLayout
            - View의 크기 및 위치를 레이아웃시킨다.
            - 다음 사이클에서 layoutSubviews()를 호출시킨다.
        2. layoutIfNeeded
            - View의 크기 및 위치를 레이아웃시킨다. (setNeedLayout과 동일)
            - 즉시 layoutSubviews()를 호출시킨다.
            - 레이아웃을 통한 애니메이션을 구현할 때 사용한다!
        3. setNeedsDisplay
            - View의 요소들을 draw한다.
            - 다음 사이클에서 draw()를 호출시킨다.
        4. displayIfNeeded
            - View의 요소들을 draw한다. (setNeedDisplay와 동일)
            - 즉시 draw()를 호출시킨다.

## GCD, OperationQueue, Swift Concurrency

- GCD
    - 가장 저수준의 동시성 프로그래밍 API
- OperationQueue
    - GCD보다 객체지향 방식인 중간 수준의 API
    - ‘작업 객체’를 관리한다.
- Swift Concurrency
    - 언어 차원의 최신 동시성 API

### GCD (Grand Central Dispatch)

- Queue에 작업을 클로저 단위로 전달해서 실행시키는 방식
    
    ```swift
    DispatchQueue.main.async {
    	// 메인 스레드 작업 (EX. UI 작업)
    }
    
    DispatchQueue.global().async {
    	// 백그라운드 작업 (EX. API 호출)
    }
    ```
    
- 특징
    - 큐 기반으로 작업 실행을 맡긴다.
    - 작업을 순차적으로 실행한다. (FIFO)
- 단점
    - 실행시킨 작업을 취소하기 어렵다.
    - 작업간의 우선순위를 관리하기 어렵다.
    - 여러개의 비동기 작업을 연결할 때, 콜백 중첩이 심해질 수 있다. (Pyramid of doom)
        
        ![image.png](attachment:aaffec5c-9736-43ad-97ff-77a8bb1e30fa:image.png)
        
    - 스레드 폭발 문제
        - `global().async` 에 다수의 작업을 던지면, 스레드를 무한정 만들어서 작업한다.
        - context switching 비용이 기하급수적으로 늘어나고, 앱이 멈추거나 느려진다.
- ‘메인 스레드에서의 UI 업데이트’ 처럼 단순하고 일회성인 작업을 실행할 때 적합하다.

### OperationQueue

- 작업을 클로저로 전달해서 시키는 것이 아니라, Operation이라는 객체로 만들어서 관리한다.
    
    ```swift
    let operationQueue = OperationQueue()
    operationQueue.maxConcurrentOperationCount = 2 // 동시에 실행할 수 있는 작업의 개수를 제한
    
    let operation = BlockOperation {
    	...
    }
    
    operationQueue.addOperations([operation], waitUntilFinished: false)
    ```
    
- 장점: 비동기 작업을 통제할 수 있다.
    - 작업 간의 우선순위를 설정할 수 있다. `operationA.addDependency(operationB)`
    - 작업의 취소/정지/재개가 쉽다.
    - 작업의 상태를 추적할 수 있다. `isExecuting` `isFinished`
    - 최대 동시 작업 개수를 제한할 수 있다. `maxConcurrentOperationCount`
- 단점
    - 작업을 객체로 만들어서 관리하기 위해, 코드가 길어진다.
- 복잡한 작업 제어가 필요할 때 적합하다

### Swift Concurrency

- 2021년 새로 소개되어 따끈따끈한, **언어 차원**에서 도입된 동시성 모델
- 작업 단위
    - Task
- async/await, actor, MainActor 등
    - async 키워드로 비동기 함수를 선언한다.
        
        ```swift
        func loadData() async throws -> String {
        	...
        }
        ```
        
    - 실행할 때는 Task 블록 안에서 호출한다.
        
        ```swift
        Task {
        	let data = try await loadData()
        	print(data)
        }
        ```

## async/await의 필요성

- 비동기 코드를 동기 코드처럼 읽을 수 있게 함으로서, 가독성을 향상한다.
    - 기존의 CGD 방식은 completion을 호출하기 위해, 코드가 분리되고 오른쪽으로 파고들어간다. 이로 인해 코드 흐름을 읽기 나쁘다.
    - async/await를 사용하면 동기 코드처럼 흐름대로 읽을 수 있다.
        
        ```swift
        func fetchData() async throws -> Data {
          let data = try await fetchData()
          return data
        }
        ```
        
- completion handler를 빼먹는 실수를 방지
- do-catch 문을 사용할 수 있다!
    
    ```swift
    Task {
        do {
            let data = try await fetchData()
            print(data)
        } catch {
            print("에러 발생!")
        }
    }
    ```
    
    - GCD 방식에서는 (Data?, Error?) 처럼 옵셔널 튜플을 이용하는 식으로 억지로 해야했다.
