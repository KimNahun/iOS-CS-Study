## 은지

**Q. 싱글톤 패턴을 만들면 어디에 저장될까요?** _(질문: 나훈)_
> **은지**: Data 영역일 것 같다.
>
> **나훈**: Data, Heap 영역 두 곳에 모두 저장되지 않을까?
>
> **정리:**
>
> **결론: Swift 싱글톤은 Heap에 저장되고, 그 참조(포인터)만 Data 영역에 저장된다.**
>
> **이유**
> - Swift의 싱글톤은 보통 `class` 타입으로 만들어짐 (`static let shared = MyClass()`)
> - **참조 타입(class)은 항상 Heap에 인스턴스가 할당됨** → 싱글톤 인스턴스 본체는 Heap에 위치
> - `static let shared` 자체는 **전역 저장소(Data 영역의 BSS/Global)** 에 위치하는 변수
> - 즉, Data 영역에는 "Heap의 어느 주소를 가리키는지"에 대한 포인터만 저장됨
>
> **메모리 구조 도식**
> ```
> [Data 영역]                [Heap 영역]
> shared (포인터) ────────▶ MyClass 인스턴스 (실제 객체)
> ```
>
> **`static let`과 lazy 초기화**
> - Swift의 `static let`은 **lazy하게 초기화** 됨 (`dispatch_once` 기반, thread-safe)
> - 앱 시작 시점이 아닌, 처음 접근하는 시점에 Heap 할당이 일어남
> - 한 번 할당되면 앱이 종료될 때까지 해제되지 않음 (참조 카운트 1 이상 유지)
>
> **struct로 싱글톤을 만든다면?**
> - 값 타입(struct)은 보통 Stack에 저장되지만, **`static` 프로퍼티는 전역 저장소(Data)에 저장됨**
> - 따라서 struct 싱글톤은 Data 영역에 값 자체가 저장됨 (Heap 사용 X)
> - 단, 일반적으로 싱글톤은 class로 구현하는 것이 관례 (참조 공유 목적이 명확)
>
> **기정님의 의문 (Data 영역 크기)**
> - Data 영역 크기는 **컴파일 타임에 결정** 됨 (전역/static 변수의 개수와 타입이 컴파일 시점에 확정)
> - 따라서 Data 영역은 런타임에 가변적이지 않음
> - 다만 `static let`의 **초기화 시점은 lazy**(런타임)이지만, 그 변수가 차지할 공간 자체는 컴파일 타임에 확보됨
>
> **한 줄 정리**: class 싱글톤 → 인스턴스는 Heap, 참조는 Data. struct 싱글톤 → 값 자체가 Data.

## 나훈

**Q1. iOS에서 멜론을 듣다가 카톡으로 넘어가면 프로세스 컨텍스트 스위칭이 어떻게 발생?** _(질문: 은지)_
> **나훈**:
> - 노래를 듣는 스레드는 계속 유지되고, 그 외 UI적인 것들은 다른 앱으로 교체되지 않을까?
> - 컨텍스트 스위칭은 A앱에서 B앱으로 정보를 넘겨주는 것이 아니라, A앱의 레지스터 같은 정보를 OS단에서 어디까지 실행되었는지 저장하고, 나중에 다시 A앱에 복귀했을 때 이 정보를 다시 불러오는 용도가 아닐까?
> - A앱에서 B앱으로 정보를 넘겨야 하는 경우(예: 위젯 같은 동일 앱 내 여러 번들)에는 1개의 앱 = 1개의 프로세스라면 정보 전달이 가능하지 않을까?
> - 잘 모르겠음. 공부 요소.
>
> **정리:**
>
> **일반적인 앱 전환**
> - A앱 → background로 내려가면서 UI 및 대부분 로직 정지(suspend)
> - OS가 실행 상태(어디까지 실행됐는지)를 저장
> - B앱 → foreground로 올라오면서 CPU 할당받고 실행
> - 앱 간 데이터 전달 없이 서로 완전히 독립된 프로세스로 동작
>
> **멜론 같은 특수한 경우**
> - A앱(멜론) → background로 내려가도 오디오 기능은 유지
> - UI 및 일반 로직은 멈추고, 오디오 관련 처리만 계속 실행
> - AVAudioSession + Background Audio 권한으로 가능
> - 카카오톡으로 넘어가도 음악은 계속 나오지만 앱 자체는 활성 상태 아님
>
> **핵심 차이**
> - 일반 앱: 전환 시 거의 모든 기능 멈춤
> - 멜론: 특정 권한(오디오)만 예외적으로 계속 동작
> - 둘 다 OS가 상태 저장 후 필요 시 복구하는 구조
>
> **(번외) App Group과 컨텍스트 스위칭**
> - 컨텍스트 스위칭은 A앱 → B앱으로 넘어갈 때 CPU 실행 상태를 저장/복구하는 과정일 뿐, 데이터 전달 기능은 아님
> - App Group은 같은 개발자 앱/위젯 간에 데이터를 공유하기 위한 "공유 저장소" 역할
> - 같은 App Group이어도 앱과 위젯은 서로 다른 프로세스라 메모리 직접 공유는 불가능
> - 데이터 교환은 항상 **A가 저장 → B가 읽는 방식**
> - 예: A앱이 UserDefaults(App Group)에 값 저장 → 위젯이 실행될 때 해당 값 읽어서 표시
>
> **한 줄 정리**: 컨텍스트 스위칭은 실행 전환, App Group은 데이터 공유 — 역할이 완전히 다르다.

**Q2. 동기/비동기와 직렬/병렬은 뭐가 다른가?**
> **나훈**:
> - **스레드를 하나만 쓰는지**에서 다르다.
> - **동기/비동기**: 현재 스레드에서 하던 작업이 오래 걸리거나 API 요청처럼 시간이 얼마나 걸릴지 모르는 작업일 때, 다른 스레드에서 진행되도록 하고 현재 스레드는 작업을 이어서 함.
> - **직렬/병렬**: 하나의 작업을 하나의 스레드에서 처리하는 것보다, 여러 개의 비슷한 작업으로 쪼개서 여러 스레드에서 동시에 처리되도록 하는 것.

**Q3. 병렬작업은 어떻게 진행하나? 비동기는 `DispatchQueue.main.async`로 하는데.**
> **나훈**:
> - GCD에서는 `DispatchQueue.global()`을 사용해 백그라운드에서 수행되도록 함.
> - OperationQueue에서는 비동기로 실행할 작업들을 Operation 객체로 만들어 큐에 넣으면, 여러 스레드에서 각 작업이 병렬로 실행됨.
> - 개인적으로 생각나는 건 `async let`.

**Q4. 뮤텍스, 세마포어, 락 — 어떤 상황에 어떤 걸 쓸 것 같은가?**
> **나훈**:
> - **세마포어**.
> - 스핀락, 뮤텍스는 같은 자원을 동시에 접근함으로써 데이터 일관성을 지키지 못하는 상황을 방지.
> - 세마포어는 여러 스레드에서 동시에 접근할 수 있게 하니까, 위에서 말한 데이터 일관성 보장에 위배되지 않나 생각했음.
>
> **Swift Concurrency 관점**
> - Swift 5.5부터 Swift Concurrency가 도입됨. 임계 영역/데이터 일관성 문제도 actor 등으로 해결 가능.
> - 작업 중 `NSLock`을 쓴 적이 있는데, Swift 일정 버전 이상에서는 해결되는 문제도 있음.
> - 버전을 올리면 `NSLock` 없이 Swift Concurrency로 해결할 수 있을 것 같음.

## 기정

**Q1. 싱글톤 패턴을 만들면 어디 저장될까요?** _(질문: 나훈)_
> **기정**: Data 영역의 크기가 바뀐다고 하면, Data 영역의 크기도 런타임에 가변적으로 결정되는 건지? 그렇다면 Data 영역도 lazy하게 작동될 수 있다면 그럴 수 있지 않을까.
>
> Data 영역의 크기가 부족해지는 걸 방지하기 위해 Data 영역 크기를 컴파일 시간에 지정해두는 게 아닐까?

**Q2. NSCache랑 Disk Cache랑 뭐가 다른가요?** _(질문: 나훈)_
> **기정**:
> - **NSCache**: 조금 더 가볍다. 앱을 종료하면 삭제됨.
> - **Disk Cache**: 종료 후에도 데이터가 유지된다.

**Q3. 어떤 상황에 NSCache, 어떤 상황에 Disk Cache를 쓸 것 같나요? 코인은 어떻게 캐시되어 있을까요?** _(질문: 나훈)_
> **기정**: 디스크일 것 같다.
>
> **나훈**: NSCache랑 Disk Cache를 상황에 따라 다르게 쓰는 것도 좋을 것 같음. 코인의 식단/상점 이미지처럼 타입에 따라 구분해서 사용.
